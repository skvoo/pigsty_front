# Проверка перед установкой ticket-defenders-express на сервер Pigsty

**Репозиторий:** https://github.com/Sharcon26/ticket-defenders-express  
**Назначение:** API для приложения Ticket Defenders (Express + Prisma): auth, тикеты, загрузка файлов в S3/MinIO.

---

## 1. Кратко о сервисе

| Параметр | Значение |
|----------|----------|
| Стек | Node.js, Express, Prisma (PostgreSQL), JWT, MinIO/S3 |
| Порт по умолчанию | **3001** |
| Зависимости | PostgreSQL, MinIO (уже есть на 104.223.25.234), JWT_SECRET, env для S3 |

**Ручки API:**  
`GET /api/health`, `POST /api/auth/login`, `POST /api/auth/signup`, `GET/POST /api/tickets`, `POST /api/upload`, `GET /api/file?key=...`.

---

## 2. Конфликты

### 2.1 Порт 3001

- В конфигурации Pigsty и в документации порт **3001 нигде не используется** (заняты: 80, 443, 3000 Grafana, 5432/6432 PG, 8000 Supabase, 8885 PgAdmin, 9000 MinIO, 9100 node_exporter и т.д.).
- **Вывод:** конфликта по порту нет, 3001 можно использовать.

### 2.2 База данных и схема

- На сервере уже есть БД **`td`** и пользователь **`tdadmin`** (pigsty.yml), бакет MinIO **`td-tickets`** и пользователь **s3user_td** — это хорошо, Express как раз рассчитан на S3/MinIO и отдельную БД.
- **Важно:** текущая схема БД `td` в проекте (**sql/td_schema.sql**) **не совпадает** со схемой, которую ожидает Prisma в ticket-defenders-express:

| Что | Текущая td (td_schema.sql) | ticket-defenders-express (Prisma) |
|-----|----------------------------|------------------------------------|
| **users** | id, email, password_hash, role, created_at | + **full_name**, **phone** |
| **tickets** | id, user_id, **file_url** (один URL), created_at | id, user_id, created_at, **status**, **json_data**, **updated_at**; файлы в отдельной таблице |
| **ticket_files** | нет | id, ticket_id, file_url, created_at (несколько файлов на тикет) |

Если поднять Express без изменений и указать `DATABASE_URL` на БД `td` с текущей схемой — Prisma будет падать (нет полей и таблицы ticket_files).

**БД `td` — под этот проект.** На сервере с БД `td` работают и другие разработчики, актуальное состояние схемы и данных мы сейчас не проверяем. При необходимости привести схему к виду Prisma позже можно использовать **`sql/td_migrate_to_prisma.sql`** (опционально).

## 3. Ресурсы (сколько «жрёт»)

- **Память:** один процесс Node.js (tsx/node) для такого API обычно **~50–150 MB RAM** в простое и при умеренной нагрузке.
- **CPU:** в простое почти не нагружает; при запросах — кратковременные всплески, для типичного числа пользователей и тикетов нагрузка небольшая.
- **Диск:** код приложения и `node_modules` — порядка сотен MB; данные — в PostgreSQL и MinIO (уже учтены в инфраструктуре).
- **Сеть:** один порт 3001, исходящие запросы к PostgreSQL (6432 или 5432) и к MinIO (9000).

**Вывод:** для сервера 104.223.25.234 (с учётом уже запущенных PG, MinIO, Supabase, Grafana и т.д.) добавление одного Node-процесса на 3001 по ресурсам приемлемо. При нехватке RAM можно ограничить процесс (systemd MemoryMax) или вынести сервис на отдельный хост позже.

## 4. Интеграция с Grafana

Мониторинг Pigsty уже собирает метрики с хоста (node_exporter, Prometheus/VictoriaMetrics, дашборды по нодам). Добавить сервис в мониторинг можно так:

| Способ | Что даёт |
|--------|----------|
| **Уже есть** | Хост 104.223.25.234 уже в мониторинге — общие CPU/RAM/диск по серверу будут видны в Grafana (в т.ч. потребление процессом Express). |
| **Проверка доступности** | Добавить в Prometheus цель для **HTTP probe** на `http://104.223.25.234:3001/api/health` (blackbox_exporter или аналог) и при падении сервиса получать алерт (например, в Alertmanager → Telegram/Email по существующей схеме). |
| **Метрики приложения** | В коде Express добавить экспорт метрик Prometheus (например, `prom-client`) и endpoint `GET /metrics`, затем добавить скрап этой цели в Prometheus — в Grafana можно будет строить дашборды по запросам, латентности, ошибкам. |

Минимально полезный шаг без правок кода — **настроить проверку /api/health** и алерт при недоступности; этого достаточно, чтобы видеть «жив ли сервис» и получать уведомления. Детальные дашборды по запросам — по желанию, через добавление /metrics в приложение.

## 5. Установка на 104.223.25.234 (выполнено)

- Репозиторий: `~/ticket-defenders-express` (клонирован с GitHub).
- Node.js: установлен через nvm (v20.20.0), путь в PATH для systemd: `/home/st/.nvm/versions/node/v20.20.0/bin`.
- Конфиг: `~/.env` в каталоге приложения (DATABASE_URL → БД `td`, PgBouncer 6432; MinIO td-tickets; JWT_SECRET сгенерирован).
- Сервис: `ticket-defenders-express.service`, включён в автозагрузку, порт **3001**.
- Проверка: `curl http://104.223.25.234:3001/api/health` → `{"ok":true}`.

Команды на сервере:
- `sudo systemctl status ticket-defenders-express` — статус
- `sudo systemctl restart ticket-defenders-express` — перезапуск
- Обновление кода: `cd ~/ticket-defenders-express && git pull && npm install && npx prisma generate && sudo systemctl restart ticket-defenders-express`

## 6. Чеклист перед установкой (для других хостов)

- [ ] Подготовить `.env`: `DATABASE_URL` (PgBouncer 6432 или PostgreSQL 5432), `JWT_SECRET` (например `openssl rand -hex 32`), переменные S3/MinIO (`S3_ENDPOINT`, `S3_BUCKET=td-tickets`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_PUBLIC_URL`, `SERVER_PUBLIC_URL=http://104.223.25.234:3001`).
- [ ] Убедиться, что на сервере доступны порты 6432 (PgBouncer) и 9000 (MinIO) с того хоста, где будет запускаться Express (104.223.25.234 — локально).
- [ ] После установки: проверить `GET http://104.223.25.234:3001/api/health` → `{"ok":true}`.
- [ ] (Опционально) Добавить в Prometheus проверку `/api/health` и правило алерта при недоступности; при желании — экспорт метрик из приложения и дашборд в Grafana.
