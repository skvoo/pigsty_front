# Рекомендуемые алерты с уведомлением в Telegram

**Принцип:** только реальные проблемы и угрозы, без шума. В Telegram — по возможности только критичное (level 0 / CRIT) и выборочные предупреждения с достаточным `for`, чтобы не срабатывать на кратковременные всплески.

## 2. Уже есть в Pigsty (node / pgsql)

В `/infra/rules/node.yml` и `pgsql.yml` уже объявлены алерты. Ниже — какие из них имеет смысл **отправлять в Telegram** и при желании слегка смягчить пороги, чтобы не было лишних срабатываний.

### 2.1 Критичные (имеет смысл слать в Telegram)

| Алерт | Условие (суть) | Зачем |
|-------|-----------------|-------|
| **NodeDown** | Нода не отдаёт метрики (node_up / job node) | Сервер недоступен или node_exporter упал. |
| **NodeFsSpaceFull** | Мало места на диске (корень или критичные FS) | Риск падения сервисов и срыва бэкапов. |
| **PostgresDown** | PostgreSQL не отвечает | БД недоступна. |
| **PgbouncerDown** | PgBouncer не отвечает | Приложения не могут подключиться к БД. |
| **PatroniDown** | Patroni не отвечает | Высокий риск потери HA/файлов. |
| **PatroniFailSafeActive** | Включён failover safeguard | Требует ручного решения. |
| **PostgresPartition** | Партиция таблицы переполнена | Критично для работы БД. |
| **PostgresXidWarpAround** | XID wrap-around | Критическая угроза целостности БД. |

Эти события — реальные инциденты или прямые угрозы, уведомления по ним ненавязчивы по смыслу (редкие и по делу).

### 2.2 Предупреждения (по желанию — только при стабильной проблеме)

Чтобы не заспамить Telegram кратковременными всплесками, для следующих алертов лучше либо не слать в Telegram, либо увеличить `for` (например, до 5–10 минут) и слать только level 1 (WARN):

| Алерт | Сейчас | Рекомендация |
|-------|--------|--------------|
| **NodeOutOfMem** | WARN, for 1m | Оставить в VM; в Telegram — только если увеличите `for` до 5m. |
| **NodeCpuHigh** | WARN, for 1m | В Telegram не слать или `for: 10m`. |
| **NodeMemSwapped** | WARN, for 5m | Можно слать в Telegram — уже 5m. |
| **NodeFsFilesFull** / **NodeFdFull** | WARN, for 1m | Имеет смысл слать только при увеличении `for` (например 5m). |
| **PostgresRestart** | INFO | В Telegram лучше не слать (информационный шум). |
| **PostgresReplicationLag** / **PostgresReplicationBreak** | WARN | Слать в Telegram только если есть реплики и это для вас критично. |

Так алерты остаются «ненавязчивыми»: в Telegram в основном критика и стабильные предупреждения.

## 3. Безопасность — ненавязчивые алерты

Не слать в Telegram каждую неудачную попытку входа или единичный скан. Имеет смысл реагировать только на:

- **Уже покрыто:** **NodeDown** — пропажа ноды/экспортера (в т.ч. после перезагрузки или блокировки).
- **Диск / OOM:** **NodeFsSpaceFull**, **NodeOutOfMem** (при увеличенном `for`) — косвенно снижают риски эксплуатации (нехватка логов, падение сервисов).

Дополнительно по безопасности (если хотите минимум «секьюрити» без шума):

| Идея | Реализация | Ненавязчивость |
|------|------------|----------------|
| Массовые неудачные логины по SSH | Алерт по метрике из journal/audit (например, `rate(ssh_failures[5m]) > N`), если такие метрики есть. | Срабатывает только при явной волне попыток, не на один раз. |
| Fail2ban забанил IP | Если есть метрика типа `fail2ban_banned_total` или аналог — алерт при росте за последние N минут. | Один алерт на факт бана, не на каждую попытку. |
| Критичный сервис пропал | Уже есть: NodeDown, PostgresDown, PgbouncerDown. | Только реальные инциденты. |

Конкретные имена метрик для Fail2ban/SSH зависят от того, выносите ли вы их в экспортер (node_exporter custom, или отдельный экспортер). Если таких метрик пока нет — добавлять в Telegram алерты по безопасности не обязательно; достаточно оставить NodeDown + диск + OOM + Postgres/PgBouncer/Patroni.

## 4. Что настроить в Alertmanager для Telegram

- **Маршрут:** все алерты с `severity: CRIT` (и при желании выборочные `severity: WARN` с длинным `for`) направлять в receiver с `telegram_configs`.
- **Группировка:** `group_by: [alertname, instance]`, `group_wait: 30s`, `repeat_interval: 4h` — чтобы не слать десятки сообщений подряд при одном и том же инциденте.
- В **Grafana** (если используете её алерты): в канале уведомлений выбрать «Telegram» и привязать к тем же правилам, где нужны уведомления.

Пример фрагмента маршрута в `alertmanager.yml` (остальное — как в docs/SERVER-MONITORING-ALERTING.md):

```yaml
route:
  receiver: email
  group_by: ['alertname', 'instance']
  group_wait: 30s
  repeat_interval: 4h
  routes:
    - match: { severity: CRIT }
      receiver: telegram
    - match: { severity: WARN }
      receiver: telegram
      # опционально: только если не хотите все WARN в Telegram — уберите или добавьте match_re на конкретные alertname
```

Так алерты остаются ненавязчивыми: в Telegram в основном критика и при необходимости — стабильные предупреждения; секьюрити покрываются через NodeDown, диск, OOM и при наличии метрик — через единичные события вроде «fail2ban сработал» или «волна SSH-неудач».
