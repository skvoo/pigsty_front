## 2. Basic Auth

- **Проверка:** запрос к `http://104.223.25.234/` и к `http://104.223.25.234/ui/d/pgsql-overview` **без** учётных данных (логин/пароль).
- **Результат:** в обоих случаях вернулся ответ **200** и контент (главная — HTML портала, /ui — страница Grafana). Запрос **401 Unauthorized** не получен.
- **Вывод:** по автоматической проверке нельзя утверждать, что Basic Auth включён и запрашивает логин: доступ мог быть без авторизации. В конфиге в репо задано `nginx_users: admin: "..."` — это должно включать Basic Auth для портала, но поведение зависит от настроек Nginx на сервере (какие location защищены).
- **Проверка пользователем:** окно запроса логина/пароля **не появляется**.
- **Решение:** настроить Basic Auth (и при необходимости исправить конфиг Nginx) **в рамках этапа безопасности** (см. [PLAN-SERVER-104.md](../PLAN-SERVER-104.md), раздел «Безопасность (в конце)» и [SECURITY-PORTAL.md](../SECURITY-PORTAL.md)).

## 3. Grafana http://104.223.25.234:3000/ui/

- Пользователь сообщил: открывается.
- По внешней проверке: по адресу **http://104.223.25.234:3000/ui/** Grafana отдаёт страницу с сообщением: *"Grafana has failed to load its application files"* (рекомендации: subpath в `root_url`, reverse proxy, браузер, перезапуск grafana-server). То есть **Grafana за subpath /ui/** может быть настроена некорректно (root_url или serve_from_sub_path). Дашборды через портал по ссылкам вида http://104.223.25.234/ui/d/pgsql-overview тоже могут упираться в эту ошибку. Имеет смысл проверить в документации Pigsty настройки Grafana при работе за Nginx (subpath) и при необходимости поправить `grafana.ini` (root_url, serve_from_sub_path) или конфиг Nginx.

## 5. Сверка конфига Pigsty (репо vs сервер)

Сравнение `pigsty.yml` в репозитории и `~/pigsty/pigsty.yml` на 104.223.25.234.

### Расхождения

| Место | В репо | На сервере |
|-------|--------|------------|
| **pgb_hba_rules (комментарий над правилом)** | `# PgBouncer (6432): сейчас открыто для теста (addr: world). План: DENY ALL + allowlist —`<br>`# убрать world, прописать только доверенные IP (бэкенды, админ). См. docs/PLAN-SERVER-104.md §4.` | `# PgBouncer (6432): доступ приложения с внешних IP (Vercel и др.)` |
| **pgb_hba_rules (title правила)** | `'app user from internet (e.g. Vercel) — тест, потом allowlist'` | `'app user from internet (e.g. Vercel)'` |

Остальное совпадает: хосты, pg_users, pg_databases, infra, etcd, app, vars (version, admin_ip, nginx_users, пароли, pg_version, pgbackrest_repo и т.д.).

### Итог по конфигу

- **Фактических отличий в работе только два:** комментарий и подпись (title) у правила PgBouncer для `dbuser_app`/`app`. Логика доступа одна и та же: `addr: world`, auth: pwd.
- Чтобы конфиг на сервере совпадал с репо (и планом по DENY ALL + allowlist), можно скопировать актуальный `pigsty.yml` из репо на сервер в `~/pigsty/pigsty.yml`. После этого при следующем применении плейбуков (например pgsql для HBA) будет использоваться обновлённый комментарий и title; само правило доступа менять не обязательно до перехода на allowlist.

## 6. Рекомендации

1. **Basic Auth:** проверить в браузере, запрашивается ли логин при открытии портала; при необходимости проверить Nginx и переприменить `infra.yml -t nginx`.
2. **Grafana /ui:** при необходимости исправить настройки subpath (root_url, serve_from_sub_path или прокси) по документации Pigsty.
3. **Конфиг:** при желании синхронизировать сервер с репо — скопировать текущий `pigsty.yml` из репо на сервер в `~/pigsty/pigsty.yml`.
4. **БД app:** при переходе на новые БД под проекты — снять использование app, затем ограничить доступ или удалить БД и пользователя dbuser_app по плану (в т.ч. убрать/изменить правило в pgb_hba_rules).
