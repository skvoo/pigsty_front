# Отчёт: некорректная настройка Patroni (clonefrom) — причина и место ошибки

**Дата:** 25.02.2026  
**Сервер:** 104.223.25.234 (pg-meta)  
**Повод:** Анализ инцидента 24.02.2026 — Patroni и PostgreSQL не принимали подключения; узел в роли Replica, состояние starting.

---

## 1. Краткое содержание

На единственном узле кластера pg-meta в конфигурации Patroni на сервере было выставлено **`clonefrom: true`**. Из-за этого Patroni постоянно поднимал инстанс как **реплику**, ожидающую клонирование с primary. В одноподной конфигурации primary нет, поэтому узел бесконечно оставался в состоянии **starting** и не принимал подключения к PostgreSQL.

**Место ошибки:** сгенерированный на сервере файл **`/etc/patroni/patroni.yml`** (секция tags / bootstrap, параметр **`clonefrom: true`**).

**Инвентарь в репозитории** (`pigsty.yml`) настроен верно: для 104.223.25.234 указано **`pg_role: primary`**. Ошибка возникла не в актуальном inventory, а при развёртывании/генерации конфигурации на сервере.

---

## 2. Симптомы (что наблюдалось)

| Проверка | Результат |
|----------|-----------|
| `systemctl status patroni` | active (running) |
| `patronictl list` | pg-meta-1: **Role Replica**, **State starting**, тег **clonefrom: true** |
| Подключение к Postgres (5432) | FATAL: the database system is shutting down / Connection refused |
| Файл `standby.signal` в `/pg/data` | Отсутствовал (данные не в режиме standby) |

То есть сервис Patroni был запущен, но поднимал PostgreSQL в режиме реплики, ожидающей клонирование с несуществующего primary.

---

## 3. Где произошла ошибка

### 3.1 Конкретное место

- **Файл на сервере:** `/etc/patroni/patroni.yml`  
- **Секция:** конфигурация Patroni (в т.ч. теги экземпляра / bootstrap).  
- **Параметр:** в блоке `tags` (или аналоге) было задано **`clonefrom: true`**.

Файл помечен как **ANSIBLE MANAGED** (генерируется плейбуками Pigsty), путь по факту: `/pg/conf/pg-meta-1.yml` → симлинк/копия в `/etc/patroni/patroni.yml`.

### 3.2 Что должно быть для single-node primary

Для единственного узла, который должен быть primary, в конфигурации Patroni на этом узле должно быть:

- **`clonefrom: false`** (или параметр отсутствует / не задаёт клонирование с другого узла).

Тогда Patroni не будет пытаться поднять инстанс как реплику и сможет занять роль Leader.

---

## 4. Причина ошибки (почему оказалось clonefrom: true)

Inventory в **текущем** репозитории корректен:

```yaml
# pigsty.yml
pg-meta:
  hosts:
    104.223.25.234: { pg_seq: 1, pg_role: primary }  # <-- primary
```

Тем не менее на сервере в сгенерированном конфиге оказалось **clonefrom: true**. Возможные причины:

### Вариант A. Развёртывание с другим inventory

- Деплой выполнялся с **другой версией** или копией `pigsty.yml` (или другого inventory), в которой:
  - узел 104.223.25.234 был описан как **replica**, или  
  - в группе pg-meta сначала была только реплика (primary добавляли позже или на другом хосте).
- В таком случае шаблон Pigsty для узла с `pg_role: replica` (или для «второго» узла) выставляет **clonefrom: true**, и на сервер попал конфиг, сгенерированный под эту роль.

### Вариант B. Порядок или состав хостов при деплое

- Кластер изначально задумывался как multi-node (primary + replica).
- Плейбук запускали так, что первым развернулся узел с ролью replica (или только он был в лимите `-l`), и для него сгенерировался конфиг с **clonefrom: true**.
- Позже primary убрали из inventory или не разворачивали, а конфиг на 104.223.25.234 не перегенерировали под роль primary.

### Вариант C. Шаблон/логика Pigsty

- В используемой версии Pigsty шаблон конфигурации Patroni по умолчанию или при определённых условиях выставляет **clonefrom: true** (например, для всех узлов с `pg_seq > 1` или для роли replica).
- Из-за ошибки в переменных, порядке выполнения или бага в шаблоне этот флаг мог быть присвоен единственному узлу, который по замыслу должен быть primary (например, если при генерации роль не подставилась или подставилась неверно).

### Вариант D. Ручное или частичное изменение

- Маловероятно, но возможно: конфиг после деплоя был вручную или другим скриптом изменён и в него добавили **clonefrom: true** без учёта single-node primary.

Итог: **ошибка не в актуальном `pigsty.yml` в репозитории, а в том, какой конфиг был сгенерирован и записан на сервер в момент развёртывания (или при последующем применении плейбуков).** То есть источник — процесс деплоя/шаблоны Pigsty и использованный при деплое inventory/лимит хостов.

---

## 5. Что было сделано для восстановления (кратко)

1. На сервере в `/etc/patroni/patroni.yml` заменено **`clonefrom: true`** → **`clonefrom: false`**.
2. Остановлен зависший/старый процесс PostgreSQL, при необходимости очищено состояние в etcd (ключи status, initialize, failsafe, history для кластера pg-meta).
3. Выполнен перезапуск Patroni; узел зарегистрировался как **Leader**, PostgreSQL принимает подключения.

Подробно шаги описаны в [RUNBOOK-PATRONI-POSTGRES-RECOVERY.md](../RUNBOOK-PATRONI-POSTGRES-RECOVERY.md) и в отчёте [REPORT-PIGSTY-CHECK-2026-02-24.md](REPORT-PIGSTY-CHECK-2026-02-24.md).

## 6. Рекомендации (как не допустить повторения)

1. **Перед каждым деплоем PGSQL** проверять, что в inventory для single-node кластера единственный хост имеет **`pg_role: primary`** (в нашем случае — 104.223.25.234 в группе pg-meta).
2. **После деплоя** на сервере проверять сгенерированный конфиг:
   ```bash
   sudo grep -E "clonefrom|pg_role" /etc/patroni/patroni.yml
   ```
   Для единственного primary ожидается **clonefrom: false** (или отсутствие clonefrom).
3. **Сразу после `./pgsql.yml` (или аналога)** выполнять:
   ```bash
   sudo -u postgres patronictl -c /etc/patroni/patroni.yml list
   ```
   Убедиться, что узел в роли **Leader** и состоянии **running**, а не Replica/starting.
4. **Закрепить в runbook/чеклисте:** для single-node после деплоя обязательно проверка `patronictl list` и при необходимости ручная правка `clonefrom` в `/etc/patroni/patroni.yml` с последующим перезапуском Patroni.
5. **Если используется свой форк или копия Pigsty:** проверить шаблоны, генерирующие `patroni.yml`, и логику установки `clonefrom` в зависимости от `pg_role` и числа узлов в кластере; при необходимости исправить шаблон, чтобы для единственного primary всегда было `clonefrom: false`.
